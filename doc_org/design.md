# 設計書 (Architecture & Design)

## 1. アプリケーションアーキテクチャ
プロジェクトは、機能ごとにクラスやファイルを分割し、小規模ながら関心の分離（Separation of Concerns）を実現している。

## 2. ファイル別・クラス別構成

### `main.py`
アプリケーションのエントリポイントであり、UI（ViewとController）の責務を持つ。
- **`App` クラス (`ctk.CTk`)**
  - メインウィンドウ。ロガーとタグマネージャーをインスタンスとして保持し、各UIコンポーネント（テキストボックス、ボタン等）の配置・イベントバインドを行う。
  - `submit_task` 関数において、データの保存処理を呼び出し、直後に `self.destroy()` で即座の終了処理を行う。
- **`AutoCompleteEntry` クラス (`ctk.CTkFrame`)**
  - タグのサジェスト入力を担う独自UIプロキシコンポーネント。
  - `tk.Frame` と `tk.Listbox` を用い、文字が入力されるたびにメインウィンドウ(`top`)内の絶対座標(`place`)で候補リストをポップアップ表示させる独自のオートコンプリートロジックをカプセル化している。

### `logger.py`
ファイルI/Oとロギングフォーマットの生成を担当。
- **`TaskLogger` クラス**
  - ドキュメントフォルダ配下（`%USERPROFILE%\Documents\log`）の生成と、現在日時に合わせた `YYYY-MM_log.md` ファイルのハンドリングを行う。
  - タグとテキストコンテンツを受け取り、要求されたMarkdownフォーマットへと変換してファイルへ追記（Append）する。

### `tags_manager.py`
タグの永続化とサジェスト検索ロジックを担当。
- **`TagsManager` クラス**
  - `tags.json` のロード・セーブを隠蔽。
  - `get_suggestions(prefix)` メソッドで、入力されたプレフィックスに基づく前方一致検索（大文字・小文字を区別しない）を行い、候補のリストを返却する。

## 3. なぜこの設計か（Design Decisions）
- **SQLiteを使用しない理由:** アプリケーションの出力結果を最終的に「プレーンなテキスト」としてプロンプトと一緒にAIに直接食わせることを目標としているため、手動のパースや変換ロジックが不要で、人間もそのまま読めるMarkdownファイルでの直接保存を採用した。
- **サジェストダイアログの `tk.Frame` + `place` 採用:** 当初見込んだWindows環境における `Toplevel`（別ウィンドウ）を用いた実装だと、Z-Index（描画順）の競合や、アプリのアクティブ・非アクティブ時の描画残りが問題になったため、同一ウィンドウ内での絶対配置によるオーバーレイへと変更しUXの安定性を高めた。
- **`submit_task()` をシングルスレッドで同期実行する理由:** ファイルI/O（`logger.write_log`、`tags_manager.add_tags`）はUIスレッド上で同期実行する設計を採用している。バックグラウンドスレッドへ分離した場合、daemonスレッドはメインループ終了と同時にkillされるため、`tags.json` のtruncate後・書き込み前にプロセスが終了する可能性（= 既存タグ全消失）がある。non-daemonスレッドでも強制終了時は同様のリスクが残る。内蔵SSD上での動作を前提とするため、I/O時間は数ms以内であり、同期実行のUX上のデメリットは許容範囲と判断した。
